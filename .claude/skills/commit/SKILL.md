## 目的

ステージング済みの差分と直近のコミットスタイルから、3 つのメッセージ候補を提示し、おすすめでコミットする。

## ステージング済みの変更内容

!`git diff --staged`

## 直近のコミットメッセージスタイル

!`git log --oneline -10`

## 制約

- `git add` / `git stage` は **絶対に実行しない**。ステージングはユーザーが行う
- ステージング済みの変更がない場合は「ステージングされた変更がありません」と伝えて終了する
- `context: fork` は付けない — 候補提示→ユーザー確認→コミット実行という対話フローが主目的のため、fork すると結果が会話に戻らない

## 出力ルール

- 候補を **3つ** 検討する
- 各候補にプレフィックスをつける（`add:` / `feat:` / `fix:` / `perf:` / `refactor:` / `chore:` など）
- 基本は日本語、技術用語は英語混在OK
- おすすめの1つ を選んで、その理由を1行で添える
- 最後に `git commit -m` を実行する

## コミットメッセージの本文（任意）

会話の文脈から以下の情報が読み取れる場合は、1行目のサマリに続けて本文を追記する：

- **背景・課題**: なぜこの変更が必要だったか（例: どんな問題が起きていたか）
- **解決方法**: どうアプローチしたか
- **採用しなかった選択肢**: 他の選択肢と、それを選ばなかった理由

情報が不明・推測になる場合は書かない。わかっている事実だけを簡潔に書く。

```
fix: 〇〇の問題を修正

背景: △△により□□が起きていた。
解決: ××でアプローチした。
不採用: ◇◇案はAという理由で見送り。
```
